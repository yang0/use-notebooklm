# 第七章：Skills 作为软件——测试、版本、依赖与分发

## 从"脚本"到"软件"

演讲者指出，Skills 正在经历一个**从简单到复杂**的演化过程：

### 当前状态
- 最简单的 Skill：一个 `skill.md` + 一些基础指令
- 多数 Skill：几分钟到几小时就能构建

### 未来状态
- 复杂的 Skill：包含可执行文件、二进制、代码、脚本、资源
- 可能需要**数周甚至数月**开发和维护
- 需要像管理**软件项目**一样管理 Skills

---

## 专业工具链的四个方向

随着 Skills 变得越来越复杂，Anthropic 认为需要建立四个方向的专业工具链：

### 方向一：测试与评估（Testing & Evaluation）

**问题**：
- 如何确保 Agent 在正确的时间加载正确的 Skill？
- 如何验证 Agent 使用 Skill 后的输出质量？
- 如何衡量 Skill 对 Agent 能力的提升？

**需要的能力**：
- ✅ **触发测试**：验证 Agent 是否在正确的任务中触发 Skill
- ✅ **输出评估**：对比使用 Skill 前后的输出质量
- ✅ **回归测试**：Skill 更新后，确保 Agent 行为仍然正确

**类比**：就像软件需要单元测试和集成测试，Skills 也需要类似的测试框架。

### 方向二：版本控制（Versioning）

**问题**：
- Skill 演化后，Agent 行为也会变化
- 如何追踪这些变化？
- 如何确保团队使用一致的 Skill 版本？

**需要的能力**：
- ✅ **版本号管理**：Semantic Versioning（语义化版本）
- ✅ **变更追踪**：清楚记录每次更新改了什么
- ✅ **回滚能力**：新版本有问题时快速回退
- ✅ **Agent 行为谱系**：追踪 Skill 版本与 Agent 行为的对应关系

**类比**：就像 npm/pip 管理依赖版本，Skills 也需要版本管理工具。

### 方向三：依赖管理（Dependencies）

**问题**：
- Skill A 可能需要 Skill B
- Skill 可能需要特定的 MCP Server
- Skill 可能需要特定的运行时环境包

**需要的能力**：
- ✅ **Skill 间依赖**：声明式地指定依赖其他 Skills
- ✅ **MCP 依赖**：声明需要的 MCP Servers
- ✅ **环境依赖**：声明需要的运行时包和工具
- ✅ **可预测性**：在不同环境中行为一致

**类比**：就像 `package.json` 声明 npm 依赖，Skills 也需要类似的依赖声明。

### 方向四：分享与分发（Sharing & Distribution）

**问题**：
- 如何分享 Skills 给团队成员？
- 如何在组织内部分发？
- 如何让社区共享 Skills？

**需要的能力**：
- ✅ **组织内部分发**：企业内部的 Skills 市场/仓库
- ✅ **团队共享**：团队成员可以订阅和使用共同的 Skills
- ✅ **社区分享**：开源 Skills 的分享平台
- ✅ **权限管理**：控制谁能访问、修改、使用 Skills

**类比**：就像 App Store 分发应用，Skills 也需要分发平台。

---

## 为什么这些很重要

### 可预测性

当 Skills 明确声明依赖时：
- 在不同环境中行为一致
- 不会因为缺少某个工具而失败
- 部署到新环境时有明确的检查清单

### 可维护性

当 Skills 有版本控制时：
- 可以追踪"什么时候、为什么"改了什么
- 团队可以协同开发和维护
- 出现问题时可以快速定位和回滚

### 可扩展性

当 Skills 可以互相依赖时：
- 可以构建复杂的 Skill 组合
- 避免重复造轮子
- 生态系统可以层级式发展

### 可协作性

当 Skills 可以分享时：
- 知识在组织内流动
- 最佳实践可以快速传播
- 社区可以共建共享

## Skill Creator Skill

演讲中提到了一个非常有趣的元概念：**Skill Creator**。

### 什么是 Skill Creator

这是一个让 Claude **自己创建 Skills** 的 Skill。

### 工作原理

1. Claude 在与用户协作中发现重复模式
2. Claude 识别出可以封装成 Skill 的知识
3. Claude 使用 Skill Creator 创建新的 Skill 文件夹
4. Claude 把脚本、指令、模板写入 Skill
5. 新 Skill 立即可用

### 意义

这标志着 Agent 从"被动使用工具"进化为"主动创造工具"：
- Agent 不仅是 Skills 的消费者
- Agent 也是 Skills 的生产者
- 形成"使用 → 学习 → 创造 → 再使用"的飞轮

---

## 持续学习（Continuous Learning）

Skills 的设计是 Anthropic 实现**持续学习**的具体步骤：

### 标准化格式的保证

当 Agent 以标准化格式（Skill 文件夹）记录知识时：
- 未来的 Agent 版本可以高效使用这些记录
- 学习成果真正"可转移"
- 不是一次性对话，而是累积性资产

### 记忆的具象化

Skills 让"记忆"这个概念变得更具体：
- 不是所有信息都记录
- 只记录**程序性知识**（procedural knowledge）
- Agent 在特定任务中可以使用的操作流程

### 动态更新

信息每天都在变化：
- 税法每年更新
- 代码规范不断演进
- 业务流程持续优化

Skills 的可修改性让 Agent 能够：
- 获取新能力
- 演化现有能力
- 废弃过时的能力

### 成本效益

演讲者强调：

> **上下文学习（in-context learning）让高频变化的信息处理变得更有成本效益。**

因为：
- 不需要重新训练模型
- 只需要更新 Skill 文件
- 所有使用该 Skill 的 Agent 立即获得更新

## 未来愿景

Barry 和 Mahesh 对未来有几个关键判断：

### 1. Skills 将成为软件

- 开发周期：从几分钟 → 数周甚至数月
- 维护成本：需要专门的团队
- 质量要求：需要测试、评估、版本控制

### 2. 技能市场将出现

- 企业内部的 Skills 市场
- 开源社区的 Skills 仓库
- 商业 Skills 的付费分发

### 3. 非技术人员将主导

- 财务、法务、HR 等专业人员创造大部分 Skills
- 开发者提供基础设施和工具
- "应用层"由领域专家主导

### 4. Agent 将成为平台

- Agent 本身越来越标准化
- 差异化来自 Skills 生态
- 就像 iPhone 的价值来自 App Store

---

## 总结

> **Skills 正在从"简单的 prompt 集合"进化为"复杂的软件系统"。**
> 
> 这个进化需要：测试、版本、依赖、分发——完整的软件工程工具链。
> 
> 当 Skills 成为软件，Agent 才真正成为平台。

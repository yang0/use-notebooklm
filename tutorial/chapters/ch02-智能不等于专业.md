# 第二章：智能不等于专业——The Brilliance Gap

## 一个经典比喻

演讲者用一个非常生动的例子来说明核心问题：

> **"谁来做你的报税？是 Mahesh（300 IQ 的数学天才），还是 Barry（经验丰富的税务专家）？"**

答案是显而易见的——**你会选择 Barry**。

为什么？因为报税需要的不是智力，而是**专业知识（Expertise）**。

---

## 当前 Agent 的"聪明缺口"

Barry 和 Mahesh 指出，今天的 LLM Agent 很像 Mahesh：

### Agent 拥有什么
- ✅ **极高的智力**：能推导、能推理、能写代码
- ✅ **广泛的通用知识**：从物理到历史，几乎无所不知
- ✅ **快速学习**：能在对话中即时理解新指令

### Agent 缺乏什么
- ❌ **前置上下文**：开始任务时缺少行业背景
- ❌ **专业知识吸收**：无法有效学习人类专家的工作方式
- ❌ **持续学习**：不会从过去的经验中积累知识，每次都在"从零开始"

演讲者总结为三个关键缺失：

| 缺失能力 | 具体表现 |
|----------|----------|
| **缺乏前置上下文** | 不了解特定业务背景、组织惯例、行业规范 |
| **无法吸收专业知识** | 专家花十年积累的经验，Agent 无法有效学习 |
| **不会随时间学习** | 今天的 Agent 和昨天的 Agent 没有区别，没有记忆 |

---

## 为什么这很严重

### 场景1：报税

Agent 理论上可以推导税法、计算税率、填写表格。但：
- 它不知道你公司的特殊税务处理惯例
- 它不理解某些行业特有的税务优化策略
- 它每年都会"忘记"去年怎么报的

结果是：Agent 花了大量时间从第一原理推导（from first principles），而 Barry 只需要按照既定流程执行。

### 场景2：企业代码规范

大型企业的开发团队可能有数千名开发者。Agent 能写代码，但：
- 不知道企业的代码风格规范
- 不理解内部框架的使用约定
- 无法复用团队积累的最佳实践

结果是：Agent 写的代码每次都需要人工审核修改，反而降低了效率。

---

## Skills 如何填补这个缺口

Skills 的设计目标就是解决上述三个缺失：

### 1. 封装前置上下文

Skill 是一个文件夹，里面可以包含：
- 行业背景说明
- 组织惯例文档
- 特定任务的执行流程

Agent 在执行任务前先读取 Skill，立刻获得所需的全部上下文。

### 2. 编码专业知识

专业知识被写成：
- **Markdown 文档**：详细操作流程
- **脚本**：可执行的代码工具
- **模板**：标准化输出格式

Agent 不需要"学习"，只需要"读取"和"执行"。

### 3. 实现持续学习

由于 Skills 是文件形式：
- Agent 今天写了一个好用的脚本 → 保存到 Skill 文件夹
- 明天执行类似任务 → 直接复用昨天的脚本
- 团队其他成员 → 也能使用这个 Skill

演讲者强调：**"Claude 第 30 天应该比第 1 天做得更好"**。Skills 让这成为可能。

---

## 核心洞察总结

> **Intelligence ≠ Expertise**
> 
> 智能让你"能做"，专业知识让你"做好"。
> 
> Agent 已经够聪明了，缺的是专业知识。Skills 就是给 Agent 装专业知识的机制。

这个洞察是整个演讲的基石。理解了这个，就能理解为什么"造 Agent"不是答案，"造 Skill"才是。
